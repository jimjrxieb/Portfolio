## THE DOCUMENTATION PACKAGE

FedRAMP authorization requires 5 key documents. Here's exactly what each is and how to create it.

### Document 1: System Security Plan (SSP)

**What it is:** The master document. Describes the entire system — architecture, data flows, and how every control is implemented.

**Template:** `templates/compliance-templates/ssp-skeleton.md`

**How to fill it out:**

```markdown
# System Security Plan — [System Name]

## 1. System Description
[What the system does, who uses it, what data it processes]

### 1.1 System Type
- Cloud Service Model: [SaaS / PaaS / IaaS]
- Deployment Model: [Public / Private / Hybrid / Community]
- FedRAMP Impact Level: [Low / Moderate / High]

### 1.2 System Boundary
[What's IN scope vs OUT of scope]

In scope:
- EKS cluster (us-east-1)
- Application containers (payments-api, checkout-api, etc.)
- RDS PostgreSQL
- S3 data buckets
- ALB/NLB load balancers

Out of scope:
- Corporate email
- Developer laptops
- Third-party SaaS tools

### 1.3 Architecture Diagram
[Include: network diagram showing VPC, subnets, security groups, K8s namespaces, data flows]

### 1.4 Data Flow
[Describe how data enters, moves through, and exits the system]

User → CloudFront → ALB → Gateway (TLS termination)
  → K8s Service → Pod (mTLS) → RDS (TLS + encrypted at rest)
  → Response back through same path

### 1.5 Interconnections
| System | Direction | Data | Protocol | Authorization |
|--------|-----------|------|----------|---------------|
| AWS CloudTrail | Outbound | Audit logs | HTTPS | AWS IAM |
| [OIDC Provider] | Inbound | Auth tokens | HTTPS | OAuth2 |

## 2. Control Implementation
[One section per control family — use the control-families templates]
```

**For each control, write:**

### Document 2: Plan of Action & Milestones (POA&M)

**What it is:** Tracker for all known vulnerabilities and compliance gaps, with remediation plans and deadlines.

**Auto-generated by:** `tools/gap-analysis.py`

**Format:**
```markdown

| ID | Weakness | Severity | Control | Discovery Date | Target Fix | Status | Milestone |
|----|----------|----------|---------|---------------|------------|--------|-----------|
| POA-001 | Missing NetworkPolicies in checkout namespace | High | SC-7 | 2026-03-01 | 2026-03-15 | In Progress | Deploy default-deny + allow rules |
| POA-002 | Base image CVE-2025-1234 (libssl) | Critical | SI-2 | 2026-03-05 | 2026-03-12 | In Progress | Update to python:3.12.6-slim |
| POA-003 | No annual IR tabletop exercise | Medium | IR-4 | 2026-03-01 | 2026-04-15 | Planned | Schedule and conduct exercise |

## Status Legend
- **In Progress**: Actively being remediated
- **Planned**: Remediation scheduled, not yet started
- **Completed**: Fixed, re-scanned, verified
- **Risk Accepted**: Documented justification, approved by AO
```

## CA-2: SECURITY ASSESSMENTS

### What "compliant" looks like
- Security assessments are performed at defined intervals
- Assessment plan exists
- Results are documented (SAR)
- Findings are tracked (POA&M)

### Workflow

```bash
# 1. Run the full FedRAMP scan
/path/to/GP-CONSULTING/07-FEDRAMP-READY/tools/run-fedramp-scan.sh \
  --target /path/to/app \
  --output ./evidence/assessment-$(date +%Y%m%d) \
  --project "client-assessment-$(date +%Y%m%d)"

# 2. The gap-analysis output IS your assessment
# - control-matrix.md = control status
# - poam.md = POA&M
# - remediation-plan.md = action items

# 3. Write the SAR using the template + scan results
# 4. Update the SSP if any controls changed
```

## CA-7: CONTINUOUS MONITORING

### What "compliant" looks like
- Monitoring runs continuously (not just at assessment time)
- Automated scanning is part of CI/CD
- Security metrics are collected and reported monthly
- Control effectiveness is validated regularly

### CI/CD as continuous monitoring

```yaml
# .github/workflows/continuous-monitoring.yml
name: Continuous Compliance Monitoring
on:
  push:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'    # Weekly Monday 6am UTC

jobs:
  compliance-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Vulnerability scan
        run: |
          trivy fs --format json --output trivy-fs.json .
          trivy image --format json --output trivy-image.json $IMAGE

      - name: SAST scan
        run: semgrep --config auto --json -o semgrep.json .

      - name: Secret scan
        run: gitleaks detect --source . --report-format json --report-path gitleaks.json

      - name: NIST mapping
        run: |
          python3 tools/scan-and-map.py --target . --output evidence/

      - name: Upload evidence
        uses: actions/upload-artifact@v4
        with:
          name: compliance-evidence-${{ github.run_number }}
          path: evidence/
          retention-days: 365

      - name: Alert on new critical findings
        if: always()
        run: |
          CRITICAL=$(cat trivy-fs.json | jq '[.Results[]?.Vulnerabilities[]? | select(.Severity=="CRITICAL")] | length')
          if [ "$CRITICAL" -gt 0 ]; then
            echo "::error::$CRITICAL new critical vulnerabilities found"
            # Send Slack notification
          fi
```

## SA-3: SECURE DEVELOPMENT LIFECYCLE

### What "compliant" looks like
- Documented SDLC with security gates
- Security is integrated at every phase (not bolted on)
- Developers are trained on secure coding

### Document your SDLC

```markdown
# Secure Development Lifecycle

## Phases and Security Activities

### 1. Design
- Threat modeling for new features
- Architecture review by security team
- Data flow analysis for PII handling

### 2. Development
- Pre-commit hooks: Gitleaks (secret detection)
- IDE integration: Semgrep (real-time SAST)
- Secure coding guidelines available to all developers

### 3. Build (CI)
- SAST: Semgrep with OWASP rules
- SCA: Trivy dependency scanning
- Secret scan: Gitleaks
- Container scan: Trivy image scanning
- IaC scan: Checkov
- Policy check: Conftest with FedRAMP rules
- Gate: Critical findings block merge

### 4. Deploy (CD)
- Admission control: Kyverno policies enforce security contexts
- ArgoCD: GitOps deployment (no manual kubectl)
- Image validation: Only signed/scanned images deploy

### 5. Operate
- Runtime monitoring: Falco
- Metrics/alerts: Prometheus + AlertManager
- Log aggregation: Fluent Bit + CloudWatch/Loki
- Continuous scanning: Weekly automated scans

### 6. Decommission
- Remove RBAC bindings
- Delete secrets from secret store
- Archive audit logs
- Update component inventory
```

## SA-11: DEVELOPER TESTING

### What "compliant" looks like
- Developers run security tests as part of their workflow
- Unit tests cover security-relevant logic (auth, input validation)
- Integration tests verify security controls
- Test results are documented

### Security test examples

```python
# test_input_validation.py — evidence for SA-11
import pytest

def test_sql_injection_blocked():
    """Verify SQL injection attempts are rejected."""
    malicious_inputs = [
        "'; DROP TABLE users; --",
        "1 OR 1=1",
        "admin'--",
    ]
    for payload in malicious_inputs:
        response = client.get(f"/api/users?id={payload}")
        assert response.status_code == 400

def test_xss_escaped():
    """Verify XSS payloads are escaped in output."""
    payload = '<script>alert("xss")</script>'
    response = client.post("/api/comments", json={"text": payload})
    assert "<script>" not in response.json()["text"]

def test_auth_required():
    """Verify unauthenticated access is denied."""
    response = client.get("/api/protected")
    assert response.status_code == 401

def test_rbac_enforced():
    """Verify users can only access their own data."""
    response = client.get("/api/users/other-user-id", headers=user_a_headers)
    assert response.status_code == 403
```
