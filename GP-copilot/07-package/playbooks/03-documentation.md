# Playbook 03: FedRAMP Documentation Package

> Derived from [GP-CONSULTING/07-FEDRAMP-READY/playbooks/08-documentation-package.md + 09-3pao-prep.md](https://github.com/jimjrxieb/GP-copilot)
> Tailored for the Portfolio application (linksmlm.com)

## What This Does

Generates the 5 key FedRAMP documents required for Authority to Operate (ATO). GP-CONSULTING auto-generates the control matrix and POA&M from scanner data. The SSP, SAR, and control narratives use fill-in-the-blank templates.

## The 5 Documents

### 1. System Security Plan (SSP)

The master document. Describes the system boundary, data flows, and how each of the 323 controls is implemented.

**For Portfolio, key sections:**
- **System boundary**: k3s cluster on dedicated server, Cloudflare Tunnel, GHCR images
- **Data flow**: User → Cloudflare → Traefik → UI/API → ChromaDB → Ollama
- **Per-control narrative**: "AC-6 is implemented via Kubernetes RBAC with scoped Roles, Gatekeeper admission blocking privileged containers, and Pod Security Standards (restricted) on all namespaces."

**Template**: `compliance-templates/ssp-skeleton.md`

### 2. Plan of Action & Milestones (POA&M)

Auto-generated by `gap-analysis.py`. Tracks open findings with remediation dates:

```
| ID | Weakness | Severity | Control | Discovery | Target Fix | Status |
|----|----------|----------|---------|-----------|------------|--------|
| POA-001 | No mTLS between pods | High | SC-8 | 2026-03-15 | 2026-04-15 | Open |
| POA-002 | No formal IR plan | Medium | IR-4 | 2026-03-15 | 2026-04-01 | Open |
```

### 3. Security Assessment Report (SAR)

Assessment results summary: tools used, finding counts, control coverage, risk posture.

**For Portfolio:**
- Tools: 8 CI scanners + Falco + kubescape + kube-bench
- Control coverage: X% MET, Y% PARTIAL, Z% MISSING
- Risk posture: LOW/MODERATE/HIGH per control family

**Template**: `compliance-templates/sar-template.md`

### 4. Control Matrix

Auto-generated by `gap-analysis.py`. One-page status of all 27 priority controls:

```
| Control | Family | Status | Evidence | Priority |
|---------|--------|--------|----------|----------|
| AC-2 | AC | PARTIAL | RBAC audit | P1 |
| AC-6 | AC | MET | Gatekeeper, PSS | P2 |
| SC-7 | SC | MET | NetworkPolicy | P1 |
```

### 5. Control Family Narratives

Per-family detailed descriptions of how controls are implemented. Templates provided for all 10 families:

- AC (Access Control) — RBAC, Gatekeeper, PSS
- AU (Audit) — K8s audit, Falco, Fluent Bit, Prometheus
- CA (Assessment) — CI/CD pipeline, gap-analysis.py
- CM (Configuration) — Git/ArgoCD, Kyverno, Checkov
- IA (Identity) — OIDC, Tailnet, External Secrets
- IR (Incident Response) — Falco alerts, response playbooks
- RA (Risk) — Trivy, Semgrep, POA&M tracking
- SA (Acquisition) — SDLC documentation, CI gates
- SC (System/Comms) — NetworkPolicy, TLS, KMS
- SI (Integrity) — Scanning, monitoring, input validation

## Evidence Freshness

For 3PAO assessment, all evidence must be <30 days old:

```bash
# Re-run full scan to refresh evidence
bash run-fedramp-scan.sh --client-name "Portfolio" --target-dir . --output-dir ./evidence-fresh

# Re-run cluster audit
bash run-cluster-audit.sh --output ./cluster-evidence

# Re-generate account inventory
kubectl get clusterrolebindings -o json > ./access-evidence/rbac.json
```

## Continuous Compliance (Post-ATO)

| Frequency | Action | Tool |
|-----------|--------|------|
| Every commit | SAST + SCA + secrets scan | GitHub Actions |
| Every deploy | Container scan + policy check | Trivy + Conftest |
| Weekly | Audit log review | Grafana dashboard |
| Monthly | Full compliance scan + report | `run-fedramp-scan.sh` |
| Quarterly | Account review + POA&M update | RBAC audit + gap-analysis.py |
| Annually | IR tabletop exercise + 3PAO re-assessment | Process |
